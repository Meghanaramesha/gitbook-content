### Where did playable ads come from?

Playable ads used to be developed by ad networks for advertisers. However, advertisers didn‚Äôt have the control to test or experiment with different creatives. Now, there are a lot of drag-and-drop tools that make it easy to develop. Alternatively, you can also easily produce them with a little bit of coding and elbow grease.

Playable ads gained popularity in [2010](https://www.adliven.com/post/the-history-of-playable-ads-past-present-and-future#:~:text=Playable%20ads%20started%20gaining%20popularity,apps%20that%20became%20playable%20ads.) and eventually became mainstream-ish by 2018. Although the concept of playable ads is widely known, [only 7% of mobile gaming app advertisers](https://martech.org/playable-ads-and-longer-campaigns-gain-traction-in-mobile-gaming/) utilize it today. More specifically, playable ads are only used in 1% of RPG ads, 13% in Hypercasual, and 9% in shooting games.

## What a playable ad looks like

There are five simple parts to a playable ad.

1. **Intro:** a brief graphic or video that captures the user‚Äôs attention
2. **Tutorial:** a quick visual explanation of how to play the game. Some apps show a quick video tutorial or use on-screen elements like arrows, prompts, or pointers.
3. **Key gameplay mechanic**: the interactive element where users can play the game.
4. **Feedback:** rewards, points, and indicators of win or loss
5. **Call-to-Action (CTA):** something to compel the user to install and continue playing

# How playable ads are created

Playable ads are programmed with HTML5, CSS, and Javascript, all of which play unique and distinct roles.

**HTML5 (HyperText Markup Language 5) is used for:**

* AdNetworks work with HTML format ‚Äì playable ads need to be packet/bundled into html
* Structures the layout of the a
* Embed images, audio, and¬† video
* Forms and interactive elements

**JavaScript is used for:**

* Interactive elements
* In-game logic
* Responding to user actions (user progress, scorekeeping, CTA)
* JavaScript plays a central role in playable ads. It typically uses the HTML <canvas> element to render the game. With the help of a JavaScript game engine, the game is drawn into the canvas, which in this case is set to fullscreen and acts as the main display for the playable. Note: The <canvas> element allows you to draw and render graphics directly. Note 2: While using a game engine isn‚Äôt strictly required, developing without one can be quite challenging (most developers prefer using one).
* For development, it‚Äôs best to use a JavaScript game engine. However, the engine‚Äôs size is crucial because playable ads have strict size limitations. For 2D game development, engines like PixiJS and Phaser are good options. For 3D games, Three.js and Babylon.js are noteworthy. While Babylon.js is excellent for development, it requires tree-shaking to fit within size limits. Tree-shaking involves extracting only the code used in your app (in the playable ad) from the library, reducing the final package size. However, not all libraries support tree-shaking, and it can be challenging, especially if you need multiple features from the library.

## MRAID 2.0

MRAID, short for Mobile Rich Media Ad Interface Definitions, is an industry-standard API developed by the Interactive Advertising Bureau (IAB) to simplify how rich media ads run inside mobile apps. It serves as a bridge between the ad creative (usually built in HTML5 and JavaScript) and the host app‚Äôs software development kit (SDK), enabling consistent interactions and ad behavior across different mobile environments.‚Äã

### What MRAID Does

MRAID allows rich media ads to perform standard actions‚Äîexpand, resize, play videos, access device orientation, or open external links‚Äîwithout developers needing to code app-specific integrations. This ‚Äúwrite once, run anywhere‚Äù standard ensures that interactive ad creatives can be displayed properly within any MRAID-compliant app, regardless of the mobile operating system or SDK.‚Äã

### Why MRAID 2.0 Is Required

Most modern ad networks require MRAID 2.0 support because it introduced essential features for full-screen, multi-part, and interstitial ad formats‚Äîcritical functions for playable and interactive ads. While MRAID 1.0 covered basic expandable banners, version 2.0 added capabilities for video playback, orientation changes, and comprehensive ad life-cycle management. These are necessary for the seamless execution of playable ads, enabling users to engage with the ad like a mini-game before installation.‚Äã

### MRAID Implementation in Ad Networks

Playable ad creatives must comply with MRAID 2.0 standards to ensure compatibility with SDKs used by ad networks like ironSource, Unity, and AppLovin. Compliance means adhering to rules such as waiting for the¬†`mraid.ready`¬†event, using secure HTTPS assets, invoking methods like¬†`mraid.open()`¬†for redirects, and packaging creatives in a single HTML file. These rules guarantee smooth user experiences and standardized performance measurement across publishers.‚Äã

In short, MRAID 2.0 is the backbone of mobile playable ad delivery, allowing advertisers to build rich, engaging ad experiences that function uniformly across multiple ad platforms and app environments. This universalization saves developers time, reduces costs, and ensures consistent user engagement across devices.

### Specifics + validation

It‚Äôs important to mention that each ad network has its own requirements. For example, they specify how the package must be wrapped, and the **Call-to-Action (CTA)** needs to be implemented specifically in the JavaScript code for each playable ad. Additionally, some ad networks, like **Mintegral**, require special JavaScript calls. For instance, the playable ad must explicitly signal (via a JavaScript function) when the game ends.

Another key aspect is the **validation process**. To put it dramatically, it‚Äôs often comparable to uploading an app to an app store. There are a number of documented requirements, but the playable ad usually needs to pass an automated validation process first, followed by a manual review by a person to ensure compliance.

## Metrics to test and measure

Chasing surface-level metrics leads to short-term gains. Focusing on long-term metrics and testing them over time drives sustainable UA. Set up in-ad events to track player behavior and fine-tune your playable ads. And like in all things UA, you need to test your creatives to see what drives the best results. Here are the no-bullshit metrics to look at.

**Engagement Rate (ER):** percentage of users who play the playable ad

(Total Engagements / Total Impressions ) x 100

**Click-through rate (CTR):** percentage of users who click on the ad in proportion to the number of users who viewed

(Clicks/Impressions) x 100

**Conversion Rate (CVR):** percentage of users who completed the call-to-action

(Conversions/Clicks) x 100

**Impressions per mille (IPM):** how many conversions per 1,000 impressions

(Conversions/Impressions) x 1000

**Completion rate:** how many users finished playing the ad. This indicates how good the ad is at engaging the user and holding their attention.

(Users who finished the ad / total users who started the ad) x 100

**Dropouts:** how many users closed the ad before finishing playing. This can suggest issues with the creative, design, or length.

Total users who started the ad ‚Äì users who finished the ad

**Time to Engage (TTE):** how long it took users to interact with the ad. This shows if the ad starts off strong and makes a good impression or not.

**% win CTR:** how many players won the playable ad and clicked through to the store. A high win CTR means you should continue letting players win and vice versa.

(user who won the playable ad and click through / total users who won the ad ) x 100

**% lose CTR:** how many players lost the playable ad and clicked through to the store

(users who lost and clicked through / total users who lost) x 100

## Dancing Unicorn Best Practices

**Keep gameplay simple.** Users should immediately understand how to interact with the ad. You‚Äôve got only seconds to hook new players. Add a challenge for a sense of reward, but don‚Äôt overcomplicate it. For hypercasual games, keep it under 15 seconds, and under 28 seconds for more complicated games.

**Make it visually striking.** The art style should match the app and grab attention. Make sure players instantly recognize the game genre, so they understand the mechanics and jump in fast.

**Let players win, but test the data first.** Use clear visual cues like pointers, arrows, circles, and fingers to show the winning move. And when they win, make them feel good about it. Sparkle confetti or make it rain coins. Make the player feel good. Start here, then test your hypothesis by measuring %win and %lose CTR.

**Combine playables with other ad formats**. Consider showing a playable after a triple-page ad or after a short video.

**Show progress** with visual progress bars, racking up points, leaderboards, or cues that indicate milestones motivate users to complete the game. The psychology of [points systems](https://theirf.org/research_post/the-psychology-of-points/#:~:text=Perceptions%20of%20Points%20Programs%3A%20The%20Psychology%20of%20Points&text=First%2C%20we%20found%20reported%20satisfaction,than%20those%20who%20do%20not.), is one that mobile games take advantage of all the time, so why not use it here?

**End with action.** Make the next step crystal clear‚Äîwhether it‚Äôs a download, sign-up, or click-through‚Äîand ensure every interaction drives toward that goal.

**Think long-term.** Avoid showing a rare or irrelevant game mechanic (looking at you, Gardenscapes). You want installs, but you also want engaged, long-term users. Don‚Äôt trick them into playing a game it‚Äôs not. High uninstall rates could ultimately hurt your ASO rankings.

**AB test**

Sure playables are expensive to ABtest, but with no code tools you can create 3-4 variations of one ad and test it on small scale before going full custom playable frenzy.

## Beginner Mistakes to Avoid

The last thing you want is to be considered shovelware. Here are costly mistakes to avoid:

1. **Avoid asshole design**

One of the oldest plays in the book is to ask players to ‚Äúplay your game‚Äù but just send you to the App Store as soon as you click on it. Some ads have fake close (x) buttons or tiny ones that make you accidentally click through to the app store to boost CTR metrics. Don‚Äôt be Gardenscapes.

2. **Low-quality apps can‚Äôt be saved**

This goes without saying, but don‚Äôt stress about making elaborate playable ads if your game is shit. Even worse, don‚Äôt mislead people into thinking your game is something it‚Äôs not.

3. **Don‚Äôt bother with apps with small target audiences**

Unless you have an unlimited budget, it‚Äôs not worth investing in developing playable ads for apps with small audiences. Unless you use no-code tools where you can create lots of playables easily and ABtest before making a decision to go full on custom playables.

**First Interaction ‚Äì Set Up Your Defense:**
